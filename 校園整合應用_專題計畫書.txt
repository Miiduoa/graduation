二、內容：

（1）摘要

近年來，大專院校陸續建置各式校務與學習相關系統，例如教務系統、學務系統、E-learning 平台、系所網站、圖書館系統等，但多半由不同單位獨立維護，導致登入方式、網址及介面設計不一致。以學生日常使用情境來看，常見問題包括：「要查課表卻找不到正確入口」、「想看公告或活動資訊必須在多個網站與社群平台來回切換」、「臨時想查餐廳營業時間或教室位置，卻缺乏集中查詢的管道」。在智慧型手機已成為主要上網工具的情況下，這種分散且缺乏整合的資訊環境，已難以滿足學生對於「即時、集中、好上手」的需求。

本專題以「校園整合應用」為題，核心動機在於從學生的實際需求出發，重新設計一個整合校園常用服務的入口。透過單一應用程式（App 或 Web），將學生最常使用的功能，例如：當日課表與教室、學校與系所公告、活動及講座資訊、校園地圖與地點查詢、餐廳菜單與營業時間等集中呈現，讓學生在最少的點擊步驟內完成查詢，而不必再記憶多組帳號密碼或收藏多個網址。同時，系上亦鼓勵將程式設計、資料庫、系統分析等課程所學運用在實務專題，本團隊希望藉由此專題，打造一個真正貼近自身校園生活的作品。

在技術面，本專題規劃採用前後端分離的系統架構。前端部分預計使用現代化 Web / App 前端框架（例如 React / React Native 或類似技術）實作元件化及響應式介面，以支援手機與桌機等不同裝置；後端部分則採用 Node.js 搭配雲端資料庫（如 Firebase / Firestore 或其他資料庫方案）提供 RESTful API，負責公告、活動、課程、餐廳等資料的存取與管理。透過清楚劃分前後端職責與資料結構，不僅有利於團隊分工，也方便日後擴充更多校園服務或更換資料來源。

綜合而言，本專題的目的為：
1. 整合校園常用資訊入口，降低學生在多個系統與網站之間切換的時間與負擔。
2. 改善使用者體驗，以簡潔、直覺且一致的介面與操作流程回應實際使用情境。
3. 建立可擴充之技術架構，使未來欲串接更多校務系統或開放資料時，能在本專題基礎上延伸。
4. 作為智慧校園發展之參考雛形，展示學生如何從需求調查出發，結合系統分析與軟體開發提出具體解決方案。

（2）進行方法及步驟

1. 本計畫採用之方法與原因

本計畫採用「需求導向」與「前後端分離」結合「模組化設計」的方式進行，主要步驟如下：

（1）需求蒐集與分析階段  
・以線上問卷及訪談方式，蒐集資管系與其他科系學生在校園生活中的常見需求與痛點，例如課表查詢、公告與活動通知、校園地圖、餐廳資訊等。  
・彙整校內既有相關網站與系統（教務系統、學務系統、系所網頁等）的功能與資訊架構，分析目前資訊分散與操作不便之處。  
・根據調查結果，選出優先實作之核心功能模組，例如：公告與通知、課表與學期行事曆、活動與社團、校園地圖、餐廳資訊、個人設定等。

（2）系統規劃與設計階段  
・繪製 Use Case 圖、系統流程圖與資料流程圖，說明不同角色（一般學生、系上管理者等）在各功能中的操作流程。  
・設計資料庫結構與主要資料表（如使用者、公告、活動、課程、餐廳、地點等），規劃欄位、關聯與索引，以兼顧查詢效率與後續維護。  
・制定 API 規格文件，包含各功能對應之 Endpoint、請求參數與回應格式，讓前後端溝通更清楚。  
・使用 Figma 或類似工具製作介面原型 (Wireframe / Prototype)，設計首頁、功能選單、公告列表與詳情、課表檢視、活動列表、餐廳資訊、個人設定等畫面，並邀請同學試用原型，根據回饋修正 UI/UX。

（3）系統開發與實作階段  
・前端：採用 JavaScript / TypeScript 搭配 React 類框架，將各畫面切分為可重複使用的元件，例如公告卡片、課表卡片、活動項目、餐廳項目等，並實作響應式版面，以兼容手機與桌機。  
・後端：以 Node.js 搭配 Express（或類似框架）建置 RESTful API，實作公告、活動、課程、餐廳等資料的新增、查詢、修改與刪除功能，並串接 Firebase / Firestore 或其他資料庫。  
・依功能模組進行分工開發，並透過 Git 版本控制管理程式碼，使用分支與 Pull Request 流程降低衝突與錯誤。

（4）測試與修正階段  
・針對關鍵功能撰寫基本單元測試（如日期與課表資料處理、公告篩選與排序等），確保邏輯正確。  
・進行前後端整合測試，確認 API 與前端畫面互動正常，並妥善處理錯誤情況（例如查無資料、網路連線中斷等）。  
・安排同學實際操作，透過回饋表紀錄使用感受與建議，依優先順序進行介面與流程調整。

（5）成果整理與文件撰寫階段  
・彙整專題執行過程，撰寫包含研究背景、動機、需求分析、系統架構、技術選用、實作過程與成果展示等內容之專題報告。  
・製作操作手冊，說明使用流程與各功能操作方式。  
・準備期末簡報與 Demo 展示（含實機操作或錄製影片），清楚呈現系統特色與未來可擴充方向。

以上方法能兼顧使用者需求與技術實作兩個層面，並透過分階段規劃減少開發風險，對四人團隊的協作也較有章法。

2. 預計可能遭遇之困難及解決途徑

（1）無法直接串接正式校務系統資料  
正式校務系統涉及帳號權限與資安問題，學生專題不易取得正式 API 或資料庫存取權限。  
解決途徑：  
・以模擬資料 (Mock Data) 或自行設計資料表方式，建置與實際情境接近的資料內容。  
・若學校或系上允許，則以不涉及個資的方式，取得部分開放資料作為展示用範例。

（2）功能項目多，開發時間有限  
校園相關功能相當多，若試圖一次全部實作，容易影響時程與品質。  
解決途徑：  
・以需求調查結果為依據，區分「核心必備功能」與「加值延伸功能」，優先完成前者。  
・透過明確分工與定期檢視進度，必要時縮減次要功能範圍，以確保主要功能品質。

（3）跨裝置及瀏覽器相容性  
不同手機、平板與瀏覽器在畫面呈現與功能支援上可能存在差異。  
解決途徑：  
・採用成熟前端框架與 UI 元件庫，減少相容性問題。  
・在開發過程中即安排多裝置與多瀏覽器測試，及早發現並修正版面與功能異常。

（4）團隊溝通與時程管理  
四位成員的課程與其他活動時間不完全一致，若缺乏良好管理容易導致進度落後。  
解決途徑：  
・以共同行事曆或簡易甘特圖規劃各階段工作與期限。  
・安排固定週會檢討進度與問題，並視情況調整分工與排程。

・以線上問卷及訪談方式，蒐集資管系與其他科系學生在校園生活中的常見需求與痛點，例如課表查詢、公告與活動通知、校園地圖、餐廳資訊等。
・彙整校內既有相關網站與系統（教務系統、學務系統、系所網頁等）的功能與資訊架構，分析目前資訊分散與操作不便之處。
・根據調查結果，選出優先實作之核心功能模組，例如：公告與通知、課表與學期行事曆、活動與社團、校園地圖、餐廳資訊、個人設定等。

（2）系統規劃與設計階段
・繪製 Use Case 圖、系統流程圖與資料流程圖，說明不同角色（一般學生、系上管理者等）在各功能中的操作流程。
・設計資料庫結構與主要資料表（如使用者、公告、活動、課程、餐廳、地點等），規劃欄位、關聯與索引，以兼顧查詢效率與後續維護。
・制定 API 規格文件，包含各功能對應之 Endpoint、請求參數與回應格式，讓前後端溝通更清楚。
・使用 Figma 或類似工具製作介面原型 (Wireframe / Prototype)，設計首頁、功能選單、公告列表與詳情、課表檢視、活動列表、餐廳資訊、個人設定等畫面，並邀請同學試用原型，根據回饋修正 UI/UX。

（3）系統開發與實作階段
・前端：採用 JavaScript / TypeScript 搭配 React 類框架，將各畫面切分為可重複使用的元件，例如公告卡片、課表卡片、活動項目、餐廳項目等，並實作響應式版面，以兼容手機與桌機。
・後端：以 Node.js 搭配 Express（或類似框架）建置 RESTful API，實作公告、活動、課程、餐廳等資料的新增、查詢、修改與刪除功能，並串接 Firebase / Firestore 或其他資料庫。
・依功能模組進行分工開發，並透過 Git 版本控制管理程式碼，使用分支與 Pull Request 流程降低衝突與錯誤。

（3-1）AR 導航功能（真實可用版）實作策略
・定位方式採「室外 GPS + IMU（加速度計/陀螺儀）+ 校園 POI 節點」混合模式，優先確保「可到達」而非僅追求視覺效果。
・室外導航以步行路網節點規劃路線，AR 僅負責方向提示與下一個轉向點，避免長距離 AR 漂移造成誤導。
・對每棟大樓入口、路口與關鍵轉彎點建立錨點資料（節點 ID、座標、朝向、可通行時間），由後端統一管理版本。
・可靠性機制：當 GPS 精度差、羅盤不穩定或追蹤信心不足時，立即降級為 2D 地圖 + 文字路徑，並提示使用者重新校正方向。
・實作範圍先聚焦校園 1~2 條高頻路線（如校門口到系館、圖書館到餐廳），完成穩定後再擴大覆蓋區域。

（3-2）AR 導航資料可信度與精度提升機制
・資料分級管理：A 級（現地實測與校正）、B 級（學校開放資料/官方地圖）、C 級（模擬資料）。AR 導航預設只使用 A/B 級資料，C 級僅供開發測試。
・現地測繪流程：以手機在固定路線多次步行採樣，記錄節點座標、轉向角、到達時間與偏移，取中位數作為正式節點。
・定期校正機制：針對施工封路、建物出入口變更與人行動線改道，建立「每月檢查 + 事件觸發更新」流程，避免資料過期。
・資料版本控管：每次路線調整皆寫入版本號、更新日期與更新人，前端可顯示資料日期，讓使用者知道導航資訊新鮮度。
・使用者回報閉環：提供「路線不準/無法到達」回報入口，回報後進入待驗證清單，經實測確認後更新節點資料。

（4）測試與修正階段
・針對關鍵功能撰寫基本單元測試（如日期與課表資料處理、公告篩選與排序等），確保邏輯正確。
・進行前後端整合測試，確認 API 與前端畫面互動正常，並妥善處理錯誤情況（例如查無資料、網路連線中斷等）。
・安排同學實際操作，透過回饋表紀錄使用感受與建議，依優先順序進行介面與流程調整。
・AR 導航專項驗收：以固定起終點進行多次實測，記錄到達成功率、平均偏移距離、錯誤轉向次數與任務完成時間，作為是否可上線之依據。
・資料可信度驗收：抽查導航節點之資料等級、更新日期與實測紀錄，若關鍵路線仍為 C 級資料，則不得宣稱為正式可用導航。

（5）成果整理與文件撰寫階段
・彙整專題執行過程，撰寫包含研究背景、動機、需求分析、系統架構、技術選用、實作過程與成果展示等內容之專題報告。
・製作操作手冊，說明使用流程與各功能操作方式。
・準備期末簡報與 Demo 展示（含實機操作或錄製影片），清楚呈現系統特色與未來可擴充方向。

以上方法能兼顧使用者需求與技術實作兩個層面，並透過分階段規劃減少開發風險，對四人團隊的協作也較有章法。

2. 預計可能遭遇之困難及解決途徑

（1）無法直接串接正式校務系統資料
正式校務系統涉及帳號權限與資安問題，學生專題不易取得正式 API 或資料庫存取權限。
解決途徑：
・以模擬資料 (Mock Data) 或自行設計資料表方式，建置與實際情境接近的資料內容。
・若學校或系上允許，則以不涉及個資的方式，取得部分開放資料作為展示用範例。
・對外說明時明確標示資料來源等級（A/B/C），避免將示範用模擬資料誤解為正式即時資料。

（2）功能項目多，開發時間有限
校園相關功能相當多，若試圖一次全部實作，容易影響時程與品質。
解決途徑：
・以需求調查結果為依據，區分「核心必備功能」與「加值延伸功能」，優先完成前者。
・透過明確分工與定期檢視進度，必要時縮減次要功能範圍，以確保主要功能品質。

（2-1）AR 導航定位易受環境影響（漂移、遮蔽、磁場干擾）
校園環境中常見樹蔭、建物遮擋與金屬干擾，可能造成 AR 方向偏移或箭頭跳動，影響導引可信度。
解決途徑：
・先定義可量測目標：到達成功率 >= 90%、平均偏移 <= 8 公尺、單次任務錯誤轉向 <= 1 次（示範路線）。
・加入方向校正流程（8 字校正、起點朝向確認）與即時品質指標（GPS 精度、方位穩定度、追蹤信心）。
・提供降級策略：品質低於門檻時自動切換 2D 導航，並保留語音/文字轉向提示，確保任務可完成。
・以校園分區逐步上線，先完成高使用率區域，透過實測資料持續調整節點與路線。

（3）跨裝置及瀏覽器相容性
不同手機、平板與瀏覽器在畫面呈現與功能支援上可能存在差異。
解決途徑：
・採用成熟前端框架與 UI 元件庫，減少相容性問題。
・在開發過程中即安排多裝置與多瀏覽器測試，及早發現並修正版面與功能異常。

（4）團隊溝通與時程管理
四位成員的課程與其他活動時間不完全一致，若缺乏良好管理容易導致進度落後。
解決途徑：
・以共同行事曆或簡易甘特圖規劃各階段工作與期限。
・安排固定週會檢討進度與問題，並視情況調整分工與排程。

圖1說明（類別圖）：
圖1為本系統資料層與擴充層之類別關係圖，重點在於統一資料存取與可擴充設計。  
・DataSource 為核心介面，定義公告、活動、餐廳、地圖等資料讀取契約。畫面層僅依賴此介面，降低耦合。  
・FirebaseSource、MockSource、HybridSource 皆實作 DataSource：  
  - FirebaseSource：連接雲端資料庫（Firestore）與後端服務。  
  - MockSource：提供開發與展示時的模擬資料。  
  - HybridSource：可依學校設定選擇真實 API 或自動回退 Mock。  
・CachedSource 以 Proxy 方式包裝 DataSource，提供快取、背景更新、請求去重等功能，提升效能與穩定性。  
・SchoolApiAdapter、BaseApiAdapter、GenericRestAdapter、NCHUAdapter 與 AdapterRegistry 構成校園 API 擴充機制，可在不改動上層畫面的情況下新增不同學校 API。

圖1對應之使用者身分與可使用功能（由架構推導）：
・一般訪客（未登入）：可瀏覽公開資訊，如公告、活動、校園地圖、餐廳菜單。  
・已登入學生：除公開資訊外，可使用個人化功能，如課表、成績、群組、訊息、通知、收藏等。  
・系統管理/資料維護者（後端）：可透過後端機制維護公告、活動、課程等資料，並提供 App/Web 查詢。  
・外部校園系統（學校 API）：透過 Adapter 介面接入，提供各校不同格式資料來源。

| 一般訪客（未登入） | 前台使用者 | 瀏覽公告、活動、校園地圖、餐廳資訊等公開資料 |
| 已登入學生 | 前台核心使用者 | 使用課表、成績、群組、訊息、通知、收藏與個人化功能 |
| 系統管理/資料維護者（後端） | 後台維運角色 | 維護公告/活動/課程等資料，提供穩定資料服務 |
| 外部校園系統（學校 API） | 外部資料提供者 | 透過 Adapter 提供各校異質資料來源，供系統整合 |

2. 循序圖（公告載入流程）

以下流程描述使用者開啟公告頁時，系統如何透過 Hook、DataSource 與快取層讀取資料；若快取過期則先回傳舊資料並於背景更新，確保使用者體驗與資料新鮮度兼顧。

```mermaid
sequenceDiagram
autonumber
actor User
participant Screen as AnnouncementsScreen
participant Hook as useAsyncList
participant DS as DataSource(Proxy: cachedSource)
participant FB as firebaseSource
participant FS as Firestore

User->>Screen: 開啟公告頁
Screen->>Hook: 初始化 load()
Hook->>DS: listAnnouncements(schoolId)

alt 快取命中且未過期
  DS-->>Hook: 回傳 cached data
else 快取命中但過期
  DS-->>Hook: 先回傳 stale data
  par 背景更新
    DS->>FB: listAnnouncements(schoolId)
    FB->>FS: query announcements
    FS-->>FB: docs
    FB-->>DS: fresh data
    DS-->>DS: setCache + notifyCacheUpdate
  end
else 快取未命中
  DS->>FB: listAnnouncements(schoolId)
  FB->>FS: query(where schoolId, orderBy publishedAt)
  FS-->>FB: docs
  FB-->>DS: mapped announcements
  DS-->>DS: setCache
  DS-->>Hook: fresh data
end

Hook-->>Screen: setItems(items)
Screen-->>User: 顯示公告列表
```

圖2說明（循序圖）：
圖2為「公告載入流程」之循序圖，描述使用者自開啟公告頁到畫面完成呈現的互動順序。  
・參與元件依序為：User、AnnouncementsScreen、useAsyncList、DataSource(cachedSource)、firebaseSource、Firestore。  
・流程起點：使用者開啟公告頁，畫面透過 useAsyncList 呼叫 listAnnouncements(schoolId)。  
・若快取命中且未過期：系統直接回傳快取資料，提升回應速度。  
・若快取命中但過期：先回傳舊資料，再於背景向 Firebase/Firestore 更新，更新後覆寫快取。  
・若快取未命中：直接查詢 Firestore，取得資料後回寫快取，再更新畫面。  
・此流程可同時兼顧三項目標：  
  1) 使用者體驗（先顯示可用資料，避免長時間空白）  
  2) 系統效能（減少重複網路請求）  
  3) 資料正確性（背景刷新保持資料新鮮度）

圖2流程步驟表（公告載入）：
| 步驟 | 參與者 | 動作內容 | 輸出結果 |
|---|---|---|---|
| 1 | User -> AnnouncementsScreen | 使用者開啟公告頁 | 觸發資料載入流程 |
| 2 | AnnouncementsScreen -> useAsyncList | 呼叫 loader 函式 | 準備向資料層取資料 |
| 3 | useAsyncList -> DataSource(cachedSource) | 呼叫 listAnnouncements(schoolId) | 進入快取判斷 |
| 4A | cachedSource | 快取命中且未過期 | 直接回傳快取資料 |
| 4B | cachedSource -> firebaseSource -> Firestore | 快取命中但過期，背景更新 | 先回傳舊資料，再更新快取 |
| 4C | cachedSource -> firebaseSource -> Firestore | 快取未命中，直接查詢 | 取得新資料並寫入快取 |
| 5 | useAsyncList -> AnnouncementsScreen | setItems(items) 更新狀態 | 畫面重新渲染 |
| 6 | AnnouncementsScreen -> User | 顯示公告列表 | 使用者看到最新可用資料 |
